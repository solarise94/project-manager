# Backlog Note: 机构匹配 DB 候选 + LLM rerank

## 状态

当前不做，不属于现阶段主线实现。

这是一个未来可选优化项，不应被后续 agent 视为“待立即落地的功能需求”。

## 当前决策

当前机构管理策略收敛为：

1. 用户侧只做主数据搜索和选择
2. 未命中时走人工复核 / 人工新增
3. 管理员可手工维护机构、别名、院区、机构合并
4. AI 只作为管理员可选预填插件，不参与用户侧自动匹配，也不参与自动建单

在这套模式下，`DB candidate recall + LLM rerank` 不是当前必需能力。

## 为什么暂缓

当前系统已经具备：

1. 机构主数据管理
2. 别名管理
3. 机构合并
4. 院区管理
5. 客户填写时的主数据搜索

因此简称、俗称、重复机构问题，现阶段更适合通过人工补别名、人工合并、定期整理来解决，而不是每次请求都引入一层 AI rerank。

## 当前问题结论

像 `浙大一院` 这类输入，之前确实说明：

1. 本地别名覆盖还不够
2. 主数据整理仍有提升空间

但这不自动推出“必须上 rerank”。

更务实的处理方式是：

1. 把高频简称补进 `OrganizationAlias`
2. 由管理员定期清理未命中或待复核机构
3. 对重复机构做人工合并

## 什么时候再考虑启用

只有在下面情况持续出现时，再重新评估：

1. 未匹配率明显偏高
2. 人工复核队列积压
3. 别名维护成本明显高于预期
4. 用户输入脏数据很多，单靠别名和模糊匹配命中率仍然不够

## 如果未来重启这个方案

再考虑做下面这条链路：

`精确匹配 -> alias 匹配 -> DB 候选召回 -> LLM 在候选中 rerank -> 返回 candidate`

注意约束：

1. LLM 只能在给定候选中排序，不能凭空命中不存在的机构
2. rerank 结果最多返回 `candidate`，不应直接提升为 `exact`
3. 这应被视为匹配优化，而不是当前主流程依赖

## 给后续 agent 的说明

如果没有新的明确业务指令，不要主动继续实现：

1. organization candidate recall
2. organization LLM rerank
3. 用户侧 AI 匹配增强

当前默认方向是：

`主数据搜索 + 人工整理 + 管理员可选 AI 预填`
