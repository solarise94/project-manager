## 多模态智能填写系统 + 邮件后置改造

### 一、多模态草稿编排器（Phase 1-3）

#### 架构

建立了"统一编排层 + 表单插件层"架构，支持文本、图片、语音三种输入方式。核心设计是"预处理与抽取分离"：

- 图片上传后先 OCR 提取文字，回填到输入框
- 语音录音后先 ASR 转写，回填到输入框
- 用户可以看到、编辑中间结果
- 点击"解析填充"时，编排器只接收纯文本

#### Provider 层

`src/lib/draft/providers/` 下抽象了四类 provider：

| Provider | 实现 | 用途 |
|----------|------|------|
| ChatProvider | MiniMax | LLM 字段抽取（两段式） |
| VisionProvider | MiniMax VLM | 图片 OCR |
| SearchProvider | MiniMax Search + Tavily fallback | 搜索补齐 |
| SpeechProvider | 腾讯云一句话识别 | 语音转写 |

腾讯云 ASR 使用 TC3-HMAC-SHA256 签名，支持 ogg/mp4/wav 原生格式，Chrome 的 webm 通过 ffmpeg 转 wav。

#### 编排器流水线

```
文本 → LLM Pass 1 抽取 → 实体解析（客户/单位 DB 匹配）→ 搜索补齐 → LLM Pass 2 定稿 → 草稿预览
```

- Pass 1：按 FormSchema 定义从文本中抽取字段 + 置信度
- 实体解析：客户名/单位名走 DB 近似匹配（normalizedName + alias），精确匹配直接绑 ID，多候选让用户在预览中选择
- 搜索补齐：仅对低置信度 + searchable 字段触发，已匹配的实体字段跳过
- Pass 2：结合搜索证据修正字段，实体字段在 Pass 2 前暂存、后恢复，避免被 LLM 打散

#### 表单 Schema

`src/lib/draft/form-schemas/` 定义了字段级元数据：

- `entityType: "organization" | "customer"` — 标记需要实体解析的字段
- `searchable: true` — 标记可搜索补齐的字段
- `normalizer: "date" | "status"` — 后处理格式化
- `enumValues` — 枚举映射（如状态字段）

当前只有 `project.create` / `project.edit` 两个 schema，后续可扩展 customer/ticket。

#### 前端

- `DraftInputPanel`：折叠式面板，文本输入 + 图片上传（OCR 后追加文本）+ 语音录音（ASR 后追加文本）
- `DraftPreview`：Dialog 展示每个字段的值、来源 Badge、置信度指示器、实体匹配状态（已匹配/候选列表/未匹配）
- `VoiceRecorder`：浏览器 MediaRecorder 录音，格式协商（ogg > mp4 > webm），录完自动上传 + ASR + 回调文本

#### 安全

- 草稿媒体文件存储在 `.draft-media/`（非公开目录），通过 fileId 引用
- Vision provider 只接受本地文件路径，拒绝 http/data: URL（防 SSRF）
- 文件在 OCR/ASR 处理后立即删除（finally 块），TTL 1 小时兜底清扫
- 上传 MIME 校验使用前缀匹配，处理 codec 参数

### 二、邮件发送后置

#### 问题

原来邮件发送嵌入在保存步骤中，SMTP 超时或失败会阻塞甚至导致保存失败。

#### 改造

所有通知邮件改为"先落库，后台发邮件"：

1. 创建 Notification 记录时设置 `emailStatus: "pending"`
2. `sendMailInBackground()` fire-and-forget 发送，成功更新为 `"sent"`，失败更新为 `"failed"` + `emailError`
3. 前端通知中心显示邮件状态（已发送/发送中/发送失败），失败时提供"重发"按钮
4. 新增 `POST /api/notifications/retry-email` 重发端点

Magic link 邮件同样改为先存 token 再后台发邮件，不再回滚 token（24 小时自然过期）。前端文案从"已发送"改为"正在发送"。

#### 涉及文件

- `prisma/schema.prisma` — Notification 加 emailStatus/emailError
- `src/lib/mail.ts` — 新增 sendMailInBackground()
- `src/lib/types.ts` — NotificationItem 加 emailStatus/emailError
- `src/components/notification-center.tsx` — 显示邮件状态 + 重发按钮
- `src/lib/reminder.ts` — 改用 sendMailInBackground
- `src/lib/representative-link.ts` — 改为先存 token 后台发邮件
- `src/app/api/projects/[id]/comments/route.ts` — 评论通知后置
- `src/app/api/projects/[id]/route.ts` — 状态变更通知后置
- `src/app/api/tickets/[id]/replies/route.ts` — 工单回复通知后置

### 三、实体自动创建 + 草稿应用安全性

#### 草稿预览中的实体创建

- 未匹配实体字段显示 checkbox "新建单位：XXX" / "新建客户：XXX"，默认勾选
- 有候选的实体字段底部增加"都不是，新建 XXX"选项
- 用户确认后，`onApply` 调用 `/api/organizations/quick-create` 和 `/api/customers` 创建主数据
- 创建失败时 toast 明确提示，不静默吞掉

#### "先规划再执行"的实体创建策略

解决了"先创建单位再发现被已有客户的单位覆盖，留下孤儿记录"的问题。

`onApply` 分四个阶段：

1. Phase 1 — 收集所有字段，分离 orgEntity / clientEntity
2. Phase 2 — 决策：如果最终客户是已有客户且自带 organizationId，跳过单位创建（`shouldCreateOrg = false`）
3. Phase 3 — 执行 org 创建/匹配（仅在 `shouldCreateOrg` 时）
4. Phase 4 — 执行 customer 创建/匹配（使用 Phase 3 产出的 orgId）

覆盖场景：
- 单位未匹配 + 客户已有且有 org → 不创建单位，用客户的 org
- 同名候选 + 用户选已有客户且有 org → 不创建单位
- 同名候选 + 用户选"新建" → 先创建单位，再用 orgId 创建客户
- 只有单位没有客户 → 正常创建单位
- 已有客户无 org → 不影响现有流程

#### 草稿应用的异步安全性

- `DraftPreview.handleApply` 改为 async，按钮显示"创建中..."并 disabled
- Dialog 在 applying 期间门控所有关闭路径（遮罩/Esc/X/取消按钮）
- `DraftInputPanel` 所有路径（fallback/直通/预览）都 `await onApply()` 后才 toast 和 cleanup
- 新建客户成功后同步 `customerOrgId`，避免"客户已有单位但前端不知道"

#### 涉及文件

- `src/components/draft-preview.tsx` — 新建 checkbox、候选列表 __create__ 选项、async handleApply、Dialog 门控
- `src/components/draft-input-panel.tsx` — onApply 支持 async、所有路径 await
- `src/app/projects/page.tsx` — 四阶段 onApply（规划→决策→org→customer）
