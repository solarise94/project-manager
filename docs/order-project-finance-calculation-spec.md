# Spec: 订单-项目-财务计算口径

## 状态

本文档是订单、项目、财务之间的金额计算 spec 草案，承接：

1. `docs/business-architecture-refactor-note.md`
2. `docs/unified-order-model-spec.md`
3. `docs/order-workflow-spec.md`

目标是固定统一订单模型落地后的财务计算口径，尤其是：

1. 订单与项目如何去重
2. 服务、商品、混合订单如何计算进度款和应收
3. 开票、到款、未开票、未到款如何定义
4. 成本和利润如何预留
5. CRM 和员工管理应读取哪些标准汇总

本文不直接要求实现。

## 核心原则

### 事实分层

系统金额事实分为：

`订单金额 = 商业事实`

`项目预算/成本 = 交付事实`

`进度款/应收/开票/回款/利润 = 财务事实`

### 财务权威来源

财务模块是金额计算权威。

其他模块只能：

1. 展示财务摘要
2. 跳转财务详情
3. 提供计算输入

不应该：

1. 在项目页单独计算完整应收
2. 在订单页单独登记回款
3. 在 CRM 页直接计算利润率
4. 在员工管理里临时拼接金额口径

## 金额类型定义

### 订单金额

`orderAmount`

来源：

1. `Order.financeAmountOverride` 存在时使用 override
2. 否则使用 `Order.totalAmount`

含义：

客户下单或成交的商业金额。

### 订单明细金额

`orderLineAmount`

来源：

1. `OrderLine.amount`

含义：

订单中每一项商品或服务的金额。

用途：

1. 支持混合订单拆分
2. 支持商品部分 100% 应收
3. 支持服务部分按项目/交付进度应收

### 项目金额

`projectAmount`

来源：

1. `Project.budgetAmount`

含义：

项目交付管理中的项目预算或服务金额。

注意：

项目金额不一定等于订单金额。

### 有效业务额

`effectiveBusinessAmount`

含义：

用于业务规模统计的去重金额。

默认：

1. 独立订单金额
2. 未被订单覆盖的项目金额
3. 或并入项目后的项目金额

不包含：

1. 已排除订单
2. 已合并来源订单
3. 已并入项目且项目已计入的订单重复金额

### 进度款

`progressReceivableAmount`

含义：

按当前业务规则，在某个时间点或周期内应确认的阶段性应收。

规则：

1. 服务立项 30%
2. 服务完成补剩余 70%
3. 商品确认/下单 100%

### 应收金额

`receivableAmount`

含义：

截至当前应向客户收取的金额。

默认：

1. 商品订单 100%
2. 未完成服务项目 30%
3. 已完成服务项目 100%
4. 独立服务订单按交付状态计算

### 已开票金额

`invoicedAmount`

来源：

1. `ProjectInvoice`
2. 统一订单发票模型或现有订单发票模型

状态：

1. 通常统计 `REQUESTED` + `ISSUED`
2. 视图中应能区分待开票和已开具
3. `CANCELLED` 不计入

### 已到款金额

`receiptAmount`

来源：

1. `FinanceReceipt`

含义：

实际收到的钱。

### 未开票金额

`uninvoicedAmount`

默认定义：

`max(receivableAmount - invoicedAmount, 0)`

说明：

发票金额可能大于应收，超额部分单独展示，不让未开票变负数。

### 已开票未付款金额

`issuedUnpaidAmount`

默认定义：

`max(issuedInvoiceAmount - invoiceAllocatedReceiptAmount, 0)`

说明：

这是发票视角，不等同于应收视角。

### 未到款金额

`outstandingAmount`

建议保留两个口径：

1. 应收未到款：`max(receivableAmount - receiptAmount, 0)`
2. 开票未到款：`max(issuedInvoiceAmount - invoiceAllocatedReceiptAmount, 0)`

页面必须明确显示是哪一种。

### 成本金额

`costAmount`

来源：

1. `Project.budgetCost`
2. 后续 `FinanceCost` 或 `OrderCost`
3. 手动补充

短期：

可以先读取 `Project.budgetCost`。

长期：

建议独立成本表，支持挂订单、项目、客户、成本类型。

### 利润金额

`profitAmount`

建议提供多个口径：

1. 业务利润：`effectiveBusinessAmount - costAmount`
2. 应收利润：`receivableAmount - costAmount`
3. 回款利润：`receiptAmount - costAmount`

默认看板建议先使用应收利润或回款利润，避免只按订单额虚高。

### 利润率

`profitRate`

建议定义：

1. `profitAmount / 对应收入基数`
2. 收入基数为 0 时返回 null，不返回 0

## 订单与项目去重规则

### 核心规则

订单和项目关联后，是否计入财务由 `OrderProjectLink.treatment` 决定。

优先级：

1. `OrderProjectLink.treatment`
2. `Order.financeTreatment`
3. 自动推导

### PROJECT_INCLUDED

含义：

订单并入项目，订单金额只展示，不单独计入应收和有效业务额。

适用：

1. 服务订单生成项目
2. 拼好鼠服务订单绑定项目
3. 多个订单汇总成一个项目

财务计算：

1. 项目应收按 `Project.budgetAmount` 和项目状态计算
2. 订单金额不重复计入
3. 订单金额可作为项目来源金额展示
4. 如存在 `allocatedAmount`，用于项目来源结构或分摊展示

### STANDALONE

含义：

订单独立计入财务，即使它有关联项目。

适用：

1. 商品订单
2. 与项目有关但不属于项目预算的追加采购
3. 人工指定独立计入的订单

财务计算：

1. 订单金额计入有效业务额
2. 商品订单 100% 应收
3. 服务订单按交付状态或独立服务规则应收

### EXCLUDED

含义：

订单排除财务。

适用：

1. 测试订单
2. 重复订单
3. 已合并来源订单
4. 不应计入业务的异常订单

财务计算：

1. 不计入有效业务额
2. 不计入应收
3. 不计入进度款
4. 可以保留发票和回款追溯，但需要异常提示

### AUTO

自动推导建议：

1. 商品订单 -> `STANDALONE`
2. 无项目关联 -> `STANDALONE`
3. 服务订单有关联项目 -> `PROJECT_INCLUDED`
4. 混合订单 -> 按明细拆分，无法拆分时提示人工确认
5. 未分类订单 -> 不自动纳入进度款，提示人工分类

## 服务、商品、混合订单规则

### 商品订单

规则：

1. 商品订单确认后默认 100% 应收
2. 商品订单不要求创建项目
3. 商品订单可关联项目，但默认 `STANDALONE`
4. 商品订单交付状态默认可设为 `DELIVERED`

周期进度款：

1. 本周确认的商品订单 -> 本周 100%
2. 本月确认的商品订单 -> 本月 100%

### 服务订单绑定项目

规则：

1. 订单本身不直接计算应收
2. 财务以项目为计算对象
3. 项目未完成 -> 30%
4. 项目完成 -> 100%

周期进度款：

1. 本周立项服务项目 -> 本周 30%
2. 本周完成服务项目 -> 本周 70%
3. 本月立项服务项目 -> 本月 30%
4. 本月完成服务项目 -> 本月 70%

注意：

如果项目在同一周期内立项并完成，本周期应确认 100%，不能只算 30% 或只算 70%。

### 独立服务订单未绑定项目

适用：

1. 小额服务不需要项目流程
2. 历史订单未补项目

规则：

1. `deliveryStatus = PENDING` -> 30%
2. `deliveryStatus = PARTIAL` -> 可先按 30%，后续如需要再引入 partial ratio
3. `deliveryStatus = DELIVERED` -> 100%
4. `deliveryStatus = WAIVED` -> 由财务设置决定，默认 100% 或 EXCLUDED 需人工选择

### 混合订单

规则：

1. 优先按 `OrderLine.category` 拆分
2. 商品明细按商品规则
3. 服务明细按服务规则
4. 无法拆分金额时，订单分类应标记 `MIXED` 并要求人工拆分

不建议：

直接把混合订单整单设为商品或服务。

## 项目应收规则

### 服务项目

服务项目默认：

1. 未完成：`budgetAmount * 0.3`
2. 已完成：`budgetAmount`

项目完成判断：

1. `status = COMPLETED`
2. 或 `progress >= 100`

完成日期优先级：

1. `StatusHistory` 中首次进入 `COMPLETED` 的时间
2. `Project.endDate`
3. 无法判断则不进入周期完成统计

### 商品项目

商品项目默认：

1. 立项后 100% 应收
2. 不等待完成状态

商品项目判断：

1. `projectType` 包含商品、产品、耗材、设备
2. 或明确枚举为 `PRODUCT`

长期建议：

项目也应有标准 `projectCategory`，不要长期依赖文本 contains。

## 周期进度款规则

### 周期定义

周期包括：

1. 本周
2. 本月
3. 本季
4. 本年

本 spec 当前重点是本周和本月。

### 服务立项 30%

计入条件：

1. 服务项目
2. 项目 `startDate` 或创建时间在周期内
3. 项目未删除
4. 项目未排除财务

金额：

`projectAmount * 0.3`

### 服务完成 70%

计入条件：

1. 服务项目
2. 项目完成日期在周期内
3. 项目未删除
4. 项目未排除财务

金额：

`projectAmount * 0.7`

### 同周期立项并完成

如果同一服务项目在同一周期内立项并完成：

金额：

`projectAmount`

说明：

这等价于 30% + 70%，但展示时可以拆成两项。

### 商品 100%

计入条件：

1. 商品订单或商品项目
2. 确认时间或立项时间在周期内
3. 未排除财务

金额：

`amount * 1.0`

## 开票规则

### 发票统计状态

建议状态口径：

1. `DRAFT`：草稿，不计入已开票
2. `REQUESTED`：已申请，计入申请中
3. `ISSUED`：已开具，计入已开票
4. `CANCELLED`：已取消，不计入

看板应区分：

1. 申请中发票金额
2. 已开具发票金额
3. 全部有效发票金额

### 单项目多发票

允许。

统计：

1. 同项目所有非取消发票求和
2. 允许低于项目应收
3. 允许等于项目应收
4. 允许高于项目应收，但标记超额

### 多订单合并开票

允许。

要求：

1. 发票必须有 coverage 或关联明细
2. 能追溯覆盖哪些订单
3. 回款可按发票、订单或项目分摊

### 发票金额大于项目金额

允许但提示。

场景：

1. 合并开票
2. 项目外追加商品
3. 人工调整

处理：

1. 不阻止
2. 标记超额
3. 要求备注或关联覆盖来源

### 发票金额低于项目金额

允许。

处理：

1. 显示未开票金额
2. 支持继续开票

## 回款规则

### 回款挂载

`FinanceReceipt` 长期建议支持：

1. `customerId`
2. `projectId`
3. `orderId`
4. `projectInvoiceId`
5. `orderInvoiceId`

迁移期还保留：

1. `externalOrderId`
2. `externalOrderInvoiceRequestId`

### 回款归属优先级

计算时优先级：

1. 绑定发票
2. 绑定项目
3. 绑定订单
4. 绑定客户

如果一笔回款同时绑定发票和项目：

1. 发票用于开票未付款计算
2. 项目用于项目回款汇总

### 一笔回款对应多张发票

长期建议新增回款分摊表：

`FinanceReceiptAllocation`

字段：

1. `receiptId`
2. `projectInvoiceId`
3. `orderInvoiceId`
4. `projectId`
5. `orderId`
6. `amount`
7. `note`

短期：

可以先限制一笔回款挂一个发票或一个项目。

## 成本与利润预留

### 短期成本

短期可用：

1. `Project.budgetCost`

缺点：

1. 不能支持订单成本
2. 不能支持多项成本
3. 不能支持成本发生日期

### 长期成本模型

建议后续新增：

`FinanceCost`

字段：

1. `customerId`
2. `projectId`
3. `orderId`
4. `amount`
5. `costType`
6. `occurredAt`
7. `remark`
8. `createdById`

### 利润计算

建议输出：

1. 订单利润
2. 项目利润
3. 客户利润
4. 代表利润贡献

但员工绩效不要直接等同利润，绩效规则另行设计。

## 标准聚合 API 建议

为了让 CRM 和员工管理不要各自拼口径，建议财务模块提供标准聚合 API。

### 财务总览

`GET /api/finance/summary`

应返回：

1. 有效业务额
2. 进度款
3. 应收
4. 已申请开票
5. 已开票
6. 已到款
7. 应收未到款
8. 开票未到款
9. 未开票
10. 成本
11. 利润

### 客户财务汇总

`GET /api/finance/customers`

每个客户返回：

1. 订单额
2. 项目额
3. 有效业务额
4. 应收
5. 已开票
6. 已到款
7. 未开票
8. 未到款
9. 成本
10. 利润率

### 项目财务汇总

`GET /api/finance/projects`

每个项目返回：

1. 项目金额
2. 关联订单金额
3. 应收
4. 已开票
5. 已到款
6. 未开票
7. 未到款
8. 成本
9. 利润

### 订单财务汇总

`GET /api/finance/orders`

每个订单返回：

1. 订单金额
2. 有效财务金额
3. 计入口径
4. 关联项目
5. 应收
6. 已开票
7. 已到款
8. 未到款

### 代表财务汇总

`GET /api/finance/representatives`

供员工管理使用。

返回：

1. 订单立项金额
2. 项目立项金额
3. 进度款
4. 已开票
5. 已到款
6. 未到款
7. 客户数
8. 订单数
9. 项目数

## CRM 联动口径

CRM 只读取摘要。

建议客户详情展示：

1. 订单总额
2. 最近订单
3. 项目总额
4. 最近项目
5. 应收
6. 已到款
7. 未到款
8. 利润率，按权限隐藏

CRM 不做：

1. 开票
2. 回款
3. 财务 override
4. 成本调整

## 员工管理口径

员工管理读取标准聚合，不直接读原始订单表。

基础指标：

1. 本周订单额
2. 本月订单额
3. 本季订单额
4. 本年订单额
5. 本周进度款
6. 本月进度款
7. 本季进度款
8. 本年进度款
9. 本周回款
10. 本月回款
11. 本季回款
12. 本年回款

归因优先级待定：

1. `Order.representativeId`
2. `Project.representativeId`
3. CRM owner
4. 客户 owner

建议后续单独写员工绩效 spec。

## 权限与敏感字段

### 财务敏感字段

以下字段应按财务权限控制：

1. 成本
2. 利润
3. 利润率
4. financeAmountOverride
5. 财务备注
6. 未到款明细

### 代表权限

当前建议：

1. 代表不进入财务模块
2. 代表如能看订单，只看摘要
3. 不显示成本、利润、override

### USER 权限

USER 应按项目 scope、CRM scope、订单 owner scope 过滤。

需要统一函数：

1. `getFinanceOrderScope`
2. `getFinanceProjectScope`
3. `getFinanceCustomerScope`
4. `getFinanceRepresentativeScope`

## 验收标准

### 去重

1. 绑定项目且 `PROJECT_INCLUDED` 的订单不重复计入
2. `STANDALONE` 订单正常计入
3. `EXCLUDED` 订单不计入
4. 已合并来源订单不计入
5. 一个项目多个订单不会重复放大有效业务额

### 应收

1. 商品订单 100% 应收
2. 未完成服务项目 30% 应收
3. 完成服务项目 100% 应收
4. 独立服务订单按交付状态应收
5. 混合订单按明细拆分

### 周期进度款

1. 本周服务立项计 30%
2. 本周服务完成计 70%
3. 本周立项并完成计 100%
4. 本月规则同理
5. 商品订单按确认时间 100%

### 开票

1. 草稿发票不计入已开票
2. 已申请和已开具可以分别统计
3. 已取消不计入
4. 超额开票有标记
5. 未完全开票有入口继续处理

### 回款

1. 回款能按订单、项目、客户汇总
2. 绑定发票的回款能减少开票未付款
3. 未绑定发票但绑定项目的回款能进入项目回款
4. 应收未到款和开票未到款口径分开

### 聚合 API

1. 财务总览使用统一计算函数
2. 客户看板使用统一计算函数
3. 项目看板使用统一计算函数
4. 订单看板使用统一计算函数
5. 员工管理不直接拼原始表

## 实施注意事项

1. 不要在多个页面复制金额计算逻辑
2. 不要把项目预算和订单金额简单相加
3. 不要把已开票未付款和应收未到款混成一个字段
4. 不要让 `OrderSourceRecord.rawJson` 参与财务计算
5. 不要把未分类订单自动计入进度款
6. 不要用文字 projectType contains 作为长期方案
7. 不要在没有 scope 的情况下聚合财务数据
