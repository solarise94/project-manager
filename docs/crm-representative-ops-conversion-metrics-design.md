# CRM 代表运营转化指标设计

## 状态

本文定义 CRM 代表运营页新增转化指标的一版产品口径与展示方案，当前仅为设计文档，不包含代码实现。

目标是在现有“代表运营”模块上，补充更贴近销售动作质量的漏斗指标，特别是：

1. `LEAD` 到有效沟通的转化情况。
2. 有效沟通到开单的转化情况。

同时需要和当前已经存在的“新客首单率”区分，避免多个指标都被笼统称为“转化率”。

---

## 设计目标

1. 把“线索是否被代表真正触达”和“触达后是否促成开单”拆成两个独立指标。
2. 保留当前已有的新客首单率，但重新命名，避免和新增指标混淆。
3. 指标口径尽量复用现有 CRM 生命周期和订单首单数据，不引入过重的额外模型。
4. 归属逻辑尽量稳定，避免客户转手后历史动作全部被覆盖。
5. UI 层优先展示“百分比 + 分子/分母”，保证可解释性。
6. 第一版优先保证归属可解释，不追求复杂归因拆分。

---

## 当前页面已有指标

当前代表运营页已有的 `30天转化率 / 90天转化率`，本质上不是 `LEAD` 转化率，也不是“沟通到开单”的转化率。

当前口径更接近：

```text
新客首单率 = 本周期新增客户中，后续产生首单的客户占比
```

即：

1. 分母：本统计周期内新进入该代表名下的客户。
2. 分子：这些客户中，历史首单时间晚于客户进入代表池时间，并在统计窗口内完成首单的客户。

因此，这个指标建议保留，但改名为：

1. `30天新客首单率`
2. `90天新客首单率`

不建议继续只显示为“30天转化率 / 90天转化率”。

---

## 建议指标体系

建议代表运营最终展示三类转化指标：

1. `线索触达率`
2. `沟通开单率`
3. `新客首单率`

三者的业务含义如下：

| 指标 | 回答的问题 |
| --- | --- |
| `线索触达率` | 分到代表手里的线索，有多少被真正沟通过？ |
| `沟通开单率` | 代表已经有效沟通过的客户，有多少最终开了首单？ |
| `新客首单率` | 本周期新增到代表名下的客户，有多少最终开了首单？ |

---

## 指标一：线索触达率

### 定义

```text
线索触达率 = 本周期从 LEAD 首次转为 CONTACTED/FOLLOWING 的客户数 / 本周期新增或分配给代表的 LEAD 客户数
```

### 分母

`leadAssignedCount`

指统计期内首次进入该代表名下、且进入时仍处于 `LEAD` 语义的客户数。

第一版明确只统计“本期首次进入代表名下的 `LEAD`”，不包含历史遗留 `LEAD` 在本期被触达的情况。

建议纳入：

1. 新建 CRM 客户档案，默认阶段为 `LEAD`。
2. 客户从客户池分配给该代表，且分配时仍未发生有效沟通。
3. 代表主动领取客户，但没有有效沟通前，仍按 `LEAD` 计入。

不建议纳入：

1. 审批通过直接生成 `CONTACTED` 的客户。
2. 已经有有效沟通历史、只是负责人转移过来的客户。
3. 历史遗留 `LEAD` 在本期被触达的情况。

### 第一版判定规则

“进入时处于 `LEAD`”在实现上不建议依赖单个当前字段直接判断，而建议采用以下规则：

1. 以客户首次关联到该代表的时点为准。
2. 如果该客户在关联到该代表之前，已经存在首次有效沟通事件，则不计入 `leadAssignedCount`。
3. 如果该客户在关联到该代表之前，不存在首次有效沟通事件，则计入 `leadAssignedCount`。

等价理解：

```text
进入代表名下时，客户尚未被任何代表完成首次有效沟通
=> 计为 LEAD 分母
```

因此，第一版更推荐用“首次有效沟通事件是否已经发生”来判定客户是否仍属于 `LEAD` 语义，而不是依赖分配时刻的 `stage` 快照。

采用这条规则的原因：

1. 与“新客首单率”的“新增”概念保持一致。
2. 避免历史遗留 `LEAD` 因多次转手造成归属争议。
3. 避免代表接手前任遗留线索后短期内被动拉低指标。
4. 存量线索清理进度如有需要，应单独设计独立指标，不混入“线索触达率”。

### 审批客户回退场景

审批通过直接生成 `CONTACTED` 的客户，默认不计入 `线索触达率` 分母。

但如果后续发生以下场景：

1. 客户被人工回退到 `LEAD`
2. 且之后由代表新增了真实有效沟通

第一版仍不建议把这类客户重新计入 `leadAssignedCount`，原因是该客户从系统来源上并不是“未触达线索”，而是“已审批建档客户被手工回退后重新运营”。

建议这类客户继续排除在 `线索触达率` 分母之外，避免人为回退阶段对刷指标造成影响。

### 分子

`leadTouchedCount`

指这些 `LEAD` 客户中，在统计期内首次发生有效沟通，并完成：

1. `LEAD -> CONTACTED`
2. `LEAD -> FOLLOWING`

任一转化的客户数。

### 推荐判定来源

第一版建议把“线索是否被触达”统一判定为“首次有效沟通事件是否发生”，而不是优先使用阶段历史。

推荐判定源：

1. `CrmInteraction.createdByUserId`
2. `CrmVisitCheckin.userId`
3. 跟进任务完成且关联的有效互动

阶段历史仍可作为辅助校验，但不建议作为第一判定源。原因：

1. `MANUAL_UPDATE` 可能写出 `LEAD -> CONTACTED/FOLLOWING`，但并不代表真实沟通。
2. 阶段历史中的 `ownerUserId` 是阶段变更时负责人，不一定等于实际完成沟通的人。
3. 用“有效沟通事件”作为单一事实来源，可以同时服务于线索触达率和沟通开单率，避免两套口径不一致。

第一版建议把 `leadTouchedCount` 定义为：

```text
统计期内，首次有效沟通事件发生，且该事件发生前客户仍属于 LEAD 语义
```

如需在实现层做辅助校验，可额外检查：

1. 是否存在 `LEAD -> CONTACTED/FOLLOWING` 阶段历史
2. 但阶段历史不作为归属主依据

### UI 命名

建议显示为：

1. `30天线索触达率`
2. `90天线索触达率`

同时显示分子/分母，例如：

```text
42% (8 / 19)
```

## 指标二：沟通开单率

### 定义

```text
沟通开单率 = 本周期发生有效沟通的客户中，在同一统计窗口内产生首单的客户数 / 本周期发生有效沟通的客户数
```

### 分母

`communicatedCustomerCount`

指统计期内发生过至少一次有效沟通的客户数。按客户去重，不按沟通次数累计。

有效沟通沿用当前 CRM 口径：

1. 电话
2. 微信
3. 邮件
4. 会议
5. 拜访
6. 转介绍
7. 完成拜访打卡
8. 跟进任务完成且关联有效互动

### 分子

`communicatedThenOrderedCount`

指这些客户中，满足以下条件的客户数：

1. 客户历史首单存在。
2. `firstOrderAt >= firstEffectiveInteractionAt`
3. `firstOrderAt` 也在当前统计窗口内

为了贴近“沟通带来的订单”，建议首单时间必须晚于或等于首次有效沟通时间。

### 口径说明

这个指标是“沟通有没有推动客户从接触走向成交”，比新客首单率更直接反映代表销售动作质量。

建议按客户维度统计，不建议按互动条数统计，否则高频沟通会稀释或放大结果。

### 窗口口径

第一版明确采用“同窗口口径”，不采用“累计最终转化口径”。

即：

1. 分母事件发生在统计窗口内。
2. 分子事件也必须发生在同一统计窗口内。

这样可以避免过去周期的沟通开单率随着时间推移持续回升，导致历史报表口径漂移。

换言之：

```text
30 天沟通开单率
= 30 天内沟通过，且也在这 30 天内完成首单的客户占比
```

### UI 命名

建议显示为：

1. `30天沟通开单率`
2. `90天沟通开单率`

同样显示分子/分母，例如：

```text
31% (12 / 39)
```

## 指标三：新客首单率

### 定义

延续当前页面已有逻辑，但统一改名：

```text
新客首单率 = 本周期新增客户中，在同一统计窗口内产生首单的客户数 / 本周期新增客户数
```

### 分母

`newCustomerCount`

沿用当前代表运营中的新客户口径：

1. 本统计周期内客户首次进入该代表名下。
2. 当前代码中的 `anchorAt` / `effective anchor` 逻辑保持不变。

### 分子

`convertedCustomerCount`

这些新增客户中，在同一统计窗口内发生首单的客户数。

第一版明确采用“同窗口口径”，不采用累计最终转化口径。

即：

1. 客户进入代表名下的事件发生在窗口内。
2. `firstOrderAt` 也必须发生在同一窗口内。

### UI 命名

建议从现在的：

1. `30天转化率`
2. `90天转化率`

改成：

1. `30天新客首单率`
2. `90天新客首单率`

---

## 三类指标的关系

可以把代表运营理解为一个三层漏斗：

```text
LEAD
-> 首次有效沟通
-> 首单
```

对应三个指标：

1. `线索触达率`：第一层漏斗
2. `沟通开单率`：第二层漏斗
3. `新客首单率`：整体漏斗结果

它们不是互斥关系，而是观察代表运营质量的三个不同切面。

## 归属规则

为了避免客户转手后历史动作归属混乱，建议统一以下归属逻辑。

第一版统一原则：

```text
凡是和“首次有效沟通”相关的指标，归属都以首次有效沟通事件的实际执行人作为主依据
```

不以阶段历史里的 `ownerUserId` 作为主归属来源。

### 线索触达率

归属到“首次有效沟通事件的实际执行人”。

原因：

1. 这个指标衡量的是谁把线索真正碰活了。
2. 不能因为后续客户转给别人，就把首次触达成绩也转走。
3. 阶段历史中的 `ownerUserId` 可能只是阶段变更时负责人，不一定是实际沟通者。

### 沟通开单率

归属到“首次有效沟通事件的实际执行人”。

原因：

1. 该指标衡量的是从沟通到首单的销售推动质量。
2. 若首单前客户发生负责人变更，后续是否要分摊属于更复杂的 attribution 问题，第一版不建议做。

### 新客首单率

继续沿用当前逻辑，归属到“客户进入代表池时的 anchor owner”。

原因：

1. 这项指标当前已经在线使用。
2. 第一版先保持兼容，避免已有报表含义突变。

---

## 统计周期建议

建议和当前代表运营页保持一致，先支持：

1. `30天`
2. `90天`

后续如有 period 筛选需求，可再扩展到：

1. 自定义日期区间
2. 本月
3. 本季度

第一版不建议一开始就做太多时间维度。

### 窗口冻结原则

第一版建议对所有新增转化指标采用“窗口内闭环”口径：

1. 分母事件发生在窗口内。
2. 分子事件也发生在窗口内。
3. 历史窗口的统计结果不应随着未来新增订单自动上升。

这样代表运营页上的 30 天、90 天指标都可以被理解为“该窗口内完成了多少漏斗推进”，而不是“窗口内启动，未来任意时点最终会不会转化”。

## API 字段建议

建议在代表运营列表和单代表详情接口中新增以下字段：

### 线索触达率

1. `leadAssignedCount30d`
2. `leadTouchedCount30d`
3. `leadTouchRate30d`
4. `leadAssignedCount90d`
5. `leadTouchedCount90d`
6. `leadTouchRate90d`

第一版建议新增辅助说明字段或在接口注释中明确：

1. `leadAssignedCount` 的判定基于“首次关联到该代表时，客户此前尚无首次有效沟通”
2. 不基于当前 `stage` 单字段直接判断

### 沟通开单率

1. `communicatedCustomerCount30d`
2. `communicatedThenOrderedCount30d`
3. `communicationToOrderRate30d`
4. `communicatedCustomerCount90d`
5. `communicatedThenOrderedCount90d`
6. `communicationToOrderRate90d`

第一版建议明确该指标口径为：

1. 分母沟通发生在窗口内
2. 首单也必须发生在同一窗口内

### 新客首单率

建议保留现有字段，但在接口文档和 UI 文案中明确语义：

1. `newCustomerCount30d`
2. `convertedCustomerCount30d`
3. `conversionRate30d`
4. `newCustomerCount90d`
5. `convertedCustomerCount90d`
6. `conversionRate90d`

需要在 API 注释或接口文档中明确：

1. `conversionRate30d = 30天新客首单率`
2. `conversionRate90d = 90天新客首单率`

后续如要彻底去歧义，可考虑把现有字段重命名为：

1. `newCustomerFirstOrderRate30d`
2. `newCustomerFirstOrderRate90d`

但第一版不强制改字段名，只建议先改 UI 文案。

第二版可考虑补充别名字段：

1. `newCustomerFirstOrderRate30d`
2. `newCustomerFirstOrderRate90d`

并逐步淘汰 `conversionRate30d/90d` 的模糊命名。

## UI 设计建议

### 代表运营列表页

页面：`/crm/representatives`

#### 顶部汇总卡片

建议调整为：

1. `总客户数`
2. `总活跃客户`
3. `整体30天线索触达率`
4. `整体30天沟通开单率`
5. `整体30天新客首单率`
6. `整体30天复购率`
7. `逾期跟进`

90 天指标不建议与 30 天并列堆满首页第一屏。建议：

1. 第一版首页首屏只展示 30 天版本。
2. 90 天版本放到代表详情页。
3. 如首页确有补充需要，可通过 tooltip 或次级统计区提供，不作为首屏主卡片。

#### 表格列建议

当前：

1. `30天转化率`
2. `90天转化率`

建议改为：

1. `30天线索触达率`
2. `30天沟通开单率`
3. `30天新客首单率`
4. `90天新客首单率`

所有百分比建议同时展示分子/分母：

### 单代表详情页

5. `90天沟通开单率`

在详情区再补充原始计数：

1. `线索数`

3. `有效沟通客户数`
4. `沟通后开单客户数`
5. `新增客户数`
6. `新增客户首单数`

## 文案建议

为了避免继续混淆，不建议页面上再使用过于笼统的“转化率”。

建议统一采用以下文案：

1. `线索触达率`
2. `沟通开单率`
3. `新客首单率`
4. `复购率`

其中：

1. “触达”表示完成了首次有效沟通。
2. “开单”表示客户产生首单。
3. “首单率”表示从新增客户角度看的成交结果。

## 第一版实现建议

第一版建议遵循“先清楚、后复杂”的原则：

1. 保留现有新客首单率逻辑，只改 UI 命名。
2. 新增 `线索触达率`。
3. 新增 `沟通开单率`。
4. 归属规则先固定，不做多代表分摊。
5. 先做 30 天和 90 天窗口，不做自定义归因模型。

## 风险与注意事项

### 1. 历史数据完整性

如果早期阶段历史表未完整记录 `LEAD -> CONTACTED/FOLLOWING`，问题相对可控，因为第一版建议主判定源使用“首次有效沟通事件”。

但仍需要确认：

1. 历史互动数据是否完整
2. 打卡自动生成的 `VISIT` interaction 是否完整
3. 跟进任务完成关联互动的数据是否完整

### 2. 客户转手归属争议

“谁的沟通促成了首单”在复杂场景下可能存在争议。第一版建议固定为“首次有效沟通事件执行人归属”，不做复杂归因拆分。

### 3. 审批直出 CONTACTED 的客户

审批通过直接生成 `CONTACTED` 的客户，不应计入 `线索触达率` 分母，否则会抬高或扭曲代表线索转化表现。

即使后续人工把这类客户回退为 `LEAD`，第一版也不建议重新纳入 `线索触达率` 分母。

### 4. 统计窗口与首单发生时间

建议统一使用：

1. 分母事件发生在窗口内
2. 分子事件也发生在窗口内

避免出现“很早沟通过的客户，本期突然开单，被重复计入每个窗口”的问题。

---

## 判定源统一建议

第一版建议尽量统一判定源，避免同一页面的两个转化指标来自两套不同事实来源。

建议采用：

1. `线索触达率`
   以首次有效沟通事件为主判定源
2. `沟通开单率`
   同样以首次有效沟通事件为主判定源
3. `阶段历史`
   主要用于辅助校验和后续审计，不作为第一归属来源

这样可以避免以下偏差：

1. 人工改阶段会被计入“线索触达”，但实际上没有真实沟通
2. 阶段变更时负责人和实际沟通者不是同一个人
3. 线索触达率和沟通开单率分母使用不同事实来源，导致漏斗上下层不能自然对齐

---

## 推荐落地顺序

1. 先统一页面文案，把现有 `conversionRate30d/90d` 改称“新客首单率”。
2. 再在 API 中补充 `线索触达率` 所需计数字段。
3. 再补 `沟通开单率` 所需计数字段。
4. 最后调整代表运营列表页和详情页展示。

---

## 已确认项

当前版本已确认：

1. `线索触达率` 第一版只统计“本期首次进入代表名下的 LEAD”，不包含历史遗留 `LEAD` 在本期被触达的情况。
2. 页面首屏只展示 30 天版本，90 天版本收进代表详情页，首页如需补充仅作为次级信息展示。
