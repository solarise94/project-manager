# Note: 机构工作流与开发日志

## 状态

本文档是说明性 note，不是实施 spec。

后续 agent 不应把这里的内容理解为“待照单实现”的功能清单，而应理解为当前方向、边界和历史决策说明。

## 当前结论

这块工作流已经从早期的“AI 建档主流程”调整为更保守、更可控的模式：

1. 用户侧优先走主数据搜索和选择
2. 机构主数据由管理员人工维护
3. 别名、院区、机构合并由管理员定期整理
4. AI 不再作为用户侧自动建档或自动匹配主流程的一部分
5. AI 只保留为管理员可选预填能力

## 机构工作流现方向

### 1. 用户侧

用户填写机构时：

1. 先查本地主数据
2. 如果能命中，就直接选择已有机构
3. 如果未命中，允许先手动填写
4. 后续是否进入人工复核，由后台当前策略控制

重点是：

用户侧不再以 “AI 搜索 -> AI 生成草稿 -> 自动发起建档” 作为默认体验。

### 2. 管理员侧

管理员负责：

1. 新增机构
2. 编辑机构
3. 维护别名
4. 维护院区
5. 合并重复机构
6. 审核机构复核任务

### 3. AI 的定位

AI 现在只是管理员工具，不是业务主链路。

允许的定位：

1. 管理员在新建机构时，点一下获取 AI 草稿
2. 管理员在复核时，用 AI 草稿辅助填写标准名、地址、别名、院区

不应继续扩展为：

1. 用户侧自动 AI 匹配
2. 用户侧一键 AI 建档
3. AI 驱动的默认建单入口
4. 没有人工确认就推进正式主数据变更

## 为什么改成这个方向

当前系统已经有：

1. 机构主数据管理
2. 别名管理
3. 院区管理
4. 机构合并
5. 客户填写时的主数据搜索

因此简称、俗称、重复机构问题，优先通过人工整理解决更稳。

相比把 AI 放在用户侧主流程中，这样的好处是：

1. 行为更可预期
2. 错误更容易追踪
3. 不会因为 AI 抽取失败影响正常录入
4. 主数据质量更适合靠管理员持续维护

## 旧方案现在怎么处理

历史上存在过一版 “机构 AI 建档 + intake + AI 草稿复核” 的设计思路。

这套思路现在不应继续被视为当前主线需求。

如果后续看到类似下面这些旧目标，不要默认继续实现：

1. 用户未命中机构时直接显示 “申请建档”
2. 搜索无结果时默认走 “AI 建档草稿”
3. 把 Tavily / MiniMax 放在用户每次机构查询的主链路里
4. 继续强化用户侧 AI intake 入口

这些都属于历史思路，不是当前默认方向。

## DevLog 当前结论

开发日志模块已经是有效的后台能力，不再属于待设计 spec。

当前定位：

1. 管理员维护 DevLog
2. 用于记录系统更新、修复、发版、公告
3. 在后台查看和管理

因此 DevLog 这部分也不需要继续按“方案设计文档”推进，除非后续有新的明确业务需求。

## 给后续 agent 的指引

没有新的明确指令时，默认遵循下面几条：

1. 不要把 AI 恢复成用户侧机构匹配主流程
2. 不要把旧的 AI intake 方案继续当作待实现 spec
3. 优先维护主数据搜索、人工复核、别名整理、机构合并这条线
4. AI 如果继续保留，只能作为管理员可选预填插件
5. DevLog 视为已落地模块，不再按“待设计功能”处理

## 未来如果重新评估

只有在下面情况持续出现时，再考虑重新讨论 AI 主流程：

1. 未匹配机构过多
2. 人工复核队列长期积压
3. 别名维护成本明显过高
4. 业务明确要求恢复面向用户的 AI intake

在此之前，当前默认方向保持不变：

`主数据搜索 + 人工整理 + 管理员可选 AI 预填`
