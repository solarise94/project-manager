# 财务页面 UI 统一与易用性改版计划

本文档给 Kimi 作为实施说明。目标是在不重写财务业务逻辑的前提下，统一财务模块 UI 表现，强化“订单是财务核心”的主路径，并分别改善桌面端和手机端体验。

## 1. 背景与目标

当前财务模块已经在业务逻辑上逐步收口到订单，包括订单开票、订单回款、订单应收、客户垫付和订单成本。但 UI 仍保留了多个历史阶段的痕迹：页面布局、筛选区、表格、卡片、空状态、操作按钮各自实现，用户很难快速判断“下一步该处理什么”。

本轮改版目标：

- 将财务入口从“功能宫格”升级为“财务工作台”，优先呈现待处理事项。
- 将订单应收、发票、回款、成本、垫付都围绕订单上下文展示。
- 保留项目发票等历史页面，但明确降级为只读归档。
- 建立一套财务模块共享 UI 组件，减少页面间视觉和交互差异。
- 桌面端优化为高效表格与批量处理体验。
- 手机端优化为卡片列表、筛选抽屉、底部操作面板，避免横向表格和小按钮堆叠。

## 2. 当前主要问题

### 2.1 信息架构问题

- `/finance` 首页同时展示 KPI、订单财务入口、历史记录入口和配置入口，但没有突出“待匹配、待开票、待回款、待退款”等工作队列。
- `/finance/project-invoices` 和 `/finance/invoice-status` 已经是历史口径，但仍以普通页面形态出现，容易让用户误以为仍是主流程。
- `/finance/order-matching`、`/finance/order-receivables`、`/finance/invoices`、`/finance/invoice-receipt-detail` 之间职责边界不够清晰。
- 客户财务详情仍混合项目、订单、项目发票、订单发票，视觉上没有明确主次。

### 2.2 UI 统一性问题

- 页面标题、返回按钮、筛选区、分页、空状态、加载态没有统一样式。
- 多数页面直接使用原生 `<table>`，列宽、行高、金额格式、操作列、hover 状态不统一。
- 移动端虽然部分页面改成卡片，但卡片结构各不相同。
- 金额展示没有统一组件，人民币符号、小数位、颜色语义不一致。
- 状态 badge 文案和颜色映射分散在各个页面中。

### 2.3 易用性问题

- 成本管理页面仍暴露 `客户ID`、`订单ID`、`项目ID` 输入，不适合真实财务人员使用。
- 订单匹配移动端卡片内同时出现 checkbox、多个状态、分类选择、客户/项目/开票按钮，操作密度过高。
- 发票工作台缺少明确的“上传真实发票”“查看订单”“筛选状态”主操作层级。
- 回款流水页只是列表查询，没有帮助用户回到订单上下文继续处理。
- 空状态只显示“暂无数据”，缺少下一步动作引导。

## 3. 目标信息架构

### 3.1 财务首页 `/finance`

首页定位为“财务工作台”，而不是纯入口页。

顶部区域：

- 页面标题：`财务工作台`
- 副标题：简短说明当前以订单为财务核心，例如“订单维度管理开票、回款、成本与垫付”
- 右侧快捷操作：`导入订单`、`订单应收`、`发票工作台`

KPI 区域：

- `有效业务额`
- `已开票`
- `已回款`
- `未回款`
- `成本`
- `利润`

待处理队列：

- `待匹配订单`
- `待申请开票`
- `已开票未回款`
- `客户垫付待退款`
- `异常/历史数据待清理`

功能入口：

- 主入口：订单应收与回款、订单发票工作台、订单匹配与开票辅助、成本管理、客户财务看板。
- 次入口：回款流水、进度款明细。
- 归档入口：历史项目发票、旧开票状态。归档入口使用灰色/虚线样式，不能和主入口等视觉权重。

### 3.2 主流程页面

主流程应围绕订单展开：

- `/finance/order-receivables`：主财务列表，展示每个订单的金额、开票、回款、未回款。
- `/finance/invoices`：订单发票工作台，查询和处理订单发票。
- `/finance/invoice-receipt-detail`：订单回款流水，只做查询和回溯。
- `/finance/costs`：成本管理，新增成本必须优先选择订单。
- `/finance/order-matching`：导入订单后的客户/项目/财务口径匹配辅助。
- `/orders/[id]?tab=finance`：唯一完整操作中心，负责新增发票、登记回款、成本、客户垫付、垫付退款。

### 3.3 历史归档页面

- `/finance/project-invoices`：历史项目发票，只读。
- `/finance/invoice-status`：历史项目口径状态，只读或后续下线。
- 页面顶部必须显示 `历史只读` banner。
- 不展示任何新建、编辑、提交、上传、回款按钮。

## 4. 共享组件计划

优先新增或重构以下组件，放在 `src/components/finance/`。

### 4.1 FinancePageHeader

用途：统一财务页标题、说明、返回链接、右侧操作。

Props 建议：

```ts
type FinancePageHeaderProps = {
  title: string;
  description?: string;
  backHref?: string;
  backLabel?: string;
  actions?: React.ReactNode;
};
```

要求：

- 桌面端标题和操作左右布局。
- 手机端标题在上、操作按钮换行或进入更多菜单。
- 所有财务子页统一使用。

### 4.2 FinanceKpiCard / FinanceKpiGrid

现有 `StatCard` 可以升级，不必完全重写。

要求：

- 支持 `default`、`warning`、`danger`、`success`、`muted`。
- 金额统一由 `MoneyText` 渲染。
- 支持小趋势/说明文字。
- 手机端横向滑动或 2 列布局，避免 4 列压缩。

### 4.3 MoneyText

用途：统一金额展示。

Props 建议：

```ts
type MoneyTextProps = {
  value: number | null | undefined;
  tone?: "default" | "income" | "expense" | "warning" | "muted";
  compact?: boolean;
  showCurrency?: boolean;
};
```

要求：

- 默认显示 `¥12,345.67`。
- 表格金额右对齐。
- 未回款为 warning，成本为 expense，回款为 income。

### 4.4 FinanceStatusBadge

用途：统一发票、回款、匹配、财务口径、历史归档状态。

状态分组：

- 发票：草稿、已申请、已开具、已取消。
- 回款：未回款、部分回款、已回款。
- 匹配：未匹配、自动匹配、人工绑定、冲突。
- 财务口径：自动、独立计入、并入项目、排除。
- 历史：历史只读、已停用。

### 4.5 FinanceToolbar

用途：统一搜索、筛选、tabs、日期范围、批量操作。

要求：

- 桌面端横向排列。
- 手机端显示搜索框 + 筛选按钮，筛选项进入 drawer/bottom sheet。
- 支持“已筛选条件 chips”和一键清空。

### 4.6 FinanceDataTable

用途：统一桌面表格。

要求：

- sticky header。
- 统一行高、hover、空状态、加载态。
- 支持右侧固定操作列。
- 支持列定义中的 `align: "left" | "right" | "center"`。
- 金额列统一使用 `MoneyText`。
- 行点击和按钮点击要避免事件冲突。

### 4.7 FinanceMobileCard

用途：统一手机端列表卡片。

结构建议：

- 顶部：主标题 + 状态 badge。
- 次行：客户/订单/日期等上下文。
- 中部：2 到 4 个关键指标。
- 底部：主操作按钮 + 更多操作。

所有移动端财务列表都使用该组件或遵循同样结构。

### 4.8 FinanceEmptyState

用途：统一空状态。

要求：

- 文案要告诉用户下一步做什么。
- 可选 `action`，例如“去订单详情新增回款”“导入订单”“清除筛选”。

### 4.9 LegacyFinanceBanner

用途：历史项目口径页面提示。

文案示例：

> 历史只读：项目发票已停用新建和编辑。新开票、回款、成本请从订单详情页操作。

## 5. 桌面端改版方案

### 5.1 财务首页

布局：

- 第一屏：Header + 关键 KPI。
- 第二屏：待处理队列，使用列表或卡片，点击直达对应筛选后的页面。
- 第三屏：功能入口，按主流程、查询、配置、历史归档分组。

KPI 建议：

- `有效业务额`
- `已开票`
- `已回款`
- `未回款`
- `成本`
- `利润`

待处理卡片建议：

- `待匹配订单`：跳 `/finance/order-matching?tab=unmatched`
- `待申请开票`：跳 `/finance/order-receivables?invoice=none`
- `已开票未回款`：跳 `/finance/order-receivables?payment=unpaid`
- `客户垫付待退款`：跳订单列表或后续垫付列表
- `历史项目数据`：跳历史只读页面

### 5.2 订单应收与回款

桌面表格列：

- 订单号
- 客户
- 订单金额
- 已开票
- 已回款
- 未回款
- 状态
- 操作

交互：

- 行点击进入 `/orders/[id]?tab=finance`。
- 操作列提供 `查看订单`。
- `未回款` 用 warning 颜色。
- 可选增加进度条：`已回款 / 已开票`。
- 筛选项：客户、订单状态、开票状态、回款状态、日期范围。

### 5.3 订单发票工作台

桌面表格列：

- 发票状态
- 购方单位
- 订单号
- 金额
- 发票类型
- 实际发票号
- 创建时间
- 操作

操作：

- `查看订单`
- `上传真实发票`
- `查看附件`
- `取消/状态更新` 仅在允许角色显示

注意：

- 不再展示“项目”作为普通 badge。历史项目发票应在历史页面。
- 如果接口仍返回历史项目发票，要加 `历史` badge 并弱化显示。

### 5.4 回款流水

桌面表格列：

- 到款日期
- 订单号
- 客户
- 金额
- 来源
- 创建人
- 备注
- 操作

操作：

- `查看订单`
- `查看客户`

说明：

- 页面只做查询，不提供新增回款。新增回款走订单详情页。
- 顶部提供日期范围、客户、订单号、来源筛选。

### 5.5 成本管理

新增成本表单改为订单优先：

- 订单搜索选择器必选或强引导。
- 选择订单后自动展示客户、项目、订单金额。
- 成本类型、金额、日期、备注。
- 从订单详情进入时锁定订单上下文，不再显示原始 ID 输入框。

列表列：

- 日期
- 类型
- 订单
- 客户
- 金额
- 备注
- 操作

### 5.6 订单匹配与开票辅助

桌面端保留表格，但降低行内操作密度。

建议：

- 顶部 tabs：待匹配、已匹配、冲突、已排除。
- 搜索和筛选放 `FinanceToolbar`。
- 批量操作栏 sticky 在表格上方，只在选择后显示。
- 行内只保留一个主按钮 `处理` 和一个次按钮 `详情`。
- 处理菜单包含：绑定客户、关联项目、开票、排除。
- 财务分类和计入口径可以行内编辑，但要统一 Select 样式。

### 5.7 客户财务详情

结构建议：

- 顶部客户 header：姓名、编号、单位、CRM 链接。
- KPI：订单额、已开票、已回款、未回款。
- 默认 tab 改为 `订单`，其次是 `开票`、`回款`、`成本`、`历史项目`。
- 项目和项目发票放到历史或辅助 tab，不要作为默认主线。
- 所有订单行点击进入订单详情页。

## 6. 手机端改版方案

### 6.1 通用规则

- 不使用横向滚动表格。
- 列表一律使用财务移动卡片。
- 筛选进入 bottom sheet 或 drawer。
- 主操作放在页面底部 sticky 区域。
- 二级操作进入 `更多` 菜单。
- 点击目标高度至少 `h-10`。
- 金额和状态优先展示，辅助信息折叠或放次行。

### 6.2 财务首页手机端

布局：

- 顶部标题。
- KPI 横向滑动卡片。
- 待处理队列以纵向卡片展示。
- 主入口用 2 列小卡片。
- 历史归档入口折叠在页面底部。

### 6.3 订单应收手机端

卡片结构：

- 第一行：订单号 + 未回款 badge。
- 第二行：客户名。
- 第三行：订单金额、已开票、已回款。
- 第四行：未回款金额 + 回款进度条。
- 底部：`查看订单`。

筛选：

- 顶部搜索订单号/客户。
- 右侧 `筛选` 按钮打开 bottom sheet。

### 6.4 发票工作台手机端

卡片结构：

- 第一行：发票状态 + 金额。
- 第二行：购方单位。
- 第三行：订单号、发票类型、创建日期。
- 底部主操作：`查看订单`。
- 若可上传：显示 `上传发票`。

### 6.5 回款流水手机端

卡片结构：

- 第一行：金额 + 到款日期。
- 第二行：客户。
- 第三行：订单号、来源。
- 第四行：备注。
- 底部：`查看订单`。

### 6.6 成本管理手机端

新增成本：

- 使用 bottom sheet。
- 第一步选择订单。
- 第二步填写金额、类型、日期、备注。
- 不展示客户ID/订单ID/项目ID文本输入。

列表卡片：

- 第一行：成本金额 + 类型 badge。
- 第二行：订单号 / 客户。
- 第三行：日期 + 备注。
- 右上角更多菜单提供删除。

### 6.7 订单匹配手机端

当前卡片操作过密，建议：

- 卡片只展示订单号、收件人、金额、匹配状态、客户/项目绑定状态。
- 底部只保留一个主按钮：`处理`。
- 点击 `处理` 打开 bottom sheet，包含：绑定客户、关联项目、开票、排除、查看详情。
- 多选批量操作放到底部 sticky bar，选择后出现。

## 8. 视觉规范建议

颜色语义：

- 收入/回款：绿色或默认强调。
- 未回款/待处理：amber。
- 成本/退款/取消：red/destructive。
- 历史只读：muted/gray/dashed border。
- 主操作：primary。

金额规范：

- 所有金额显示 `¥`。
- 表格金额右对齐。
- KPI 金额可使用较大字号。
- 列表中金额避免过多小数噪声，但保持两位小数。

Badge 规范：

- 同一状态在所有页面颜色一致。
- Badge 文案使用中文，不直接展示枚举值。
- 历史/停用状态统一使用 `outline` 或 `secondary`。

页面宽度：

- 财务子页统一 `max-w-7xl mx-auto p-4 md:p-6`。
- 表格卡片统一圆角和边距。
- 手机端避免 `p-6` 造成内容过窄。

## 9. 交互规范建议

筛选：

- 桌面端筛选横向展示。
- 手机端筛选进 bottom sheet。
- 筛选结果显示 chips。
- 提供清除筛选。

列表：

- 桌面端行点击打开详情。
- 右侧操作列只保留最常用动作。
- 次要动作放进更多菜单。
- 手机端卡片主操作最多 1 个，其他放 bottom sheet。

表单：

- 新增成本、回款、垫付退款都必须有订单上下文。
- 用户不应手动输入数据库 ID。
- 从订单详情进入时，表单应显示订单信息且锁定上下文。

空状态：

- 不只写“暂无数据”。
- 示例：`暂无发票记录。请进入订单详情页创建订单发票。`

## 10. 验收标准

功能不回退：

- 新建开票仍只能从订单详情页走订单发票。
- 回款仍只能关联订单。
- 成本新增不破坏现有 API scope。
- 销售角色只读范围不扩大。
- 历史项目发票仍可查看但不可写。

桌面端验收：

- 财务首页能一眼看到 KPI 和待处理事项。
- 主列表页面表格风格一致。
- 金额、状态、操作列位置一致。
- 不再出现客户ID/订单ID/项目ID裸输入作为主要操作方式。
- 历史页面视觉明显弱化。

手机端验收：

- 财务主页面无横向滚动。
- 主列表均使用卡片。
- 订单匹配卡片不再堆 3 个以上操作按钮。
- 筛选可以在 bottom sheet 中完成。
- 新增成本/回款相关操作使用订单上下文。

工程验收：

- `npm run lint` 通过。
- `npm run build` 通过。
- 涉及 `useSearchParams()` 的页面继续保持 `Suspense` 包裹。
- 不修改财务 API 数据口径，除非单独说明。
- 不引入新的全局样式冲突。
